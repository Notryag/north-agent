# Plan Principles

## 核心前提

> North Agent 是产品无关的 Agent Runtime；路线由真实宿主需求推动，不由参考项目的功能清单推动。

## 判断一项工作是否进入计划

每项新工作应回答：

1. 它解决了哪个可复现的宿主问题或任务失败？
2. 该问题是否属于模型、工具、Skill、运行状态、持久化或可观测性的通用边界？
3. 它能否留在宿主实现，而不扩大 North 的长期兼容面？
4. 它是否需要新增顶层 API、状态字段、持久化协议或安全边界？
5. 如何用不依赖真实模型随机性的测试证明契约成立？

只有前两项明确，并且第三项证明应进入 Runtime，才创建 Active TODO。

## 工程原则

- 保持 `app -> north` 单向依赖
- 公开顶层 API 少而稳定，内部装配函数不承诺长期兼容
- Harness 不拥有用户、租户、权限、产品会话、计费或界面
- 宿主显式注入路径、工具、Checkpointer、事件接收器和基础设施
- 文件内容、工具输出和外部页面均按不可信数据处理
- 持久化、取消、并发、幂等和安全边界的测试优先级高于外围功能
- 不为未来假设提前引入 Sandbox、Subagent、MCP 或大而全注册系统

## 已验证闭环

- Web 调研 -> 信息整理 -> Markdown 报告 -> Artifact
- 文件输入 -> 文件发现 -> 文件读取 -> 直接回答或 Markdown 报告 -> Artifact

已验证闭环应保持稳定。新任务不需要继续扩充它们，除非宿主暴露可复现缺陷。

## 测试原则

- 小改动默认做静态检查和差异审查
- 共享 Runtime 契约运行最小受影响测试
- 持久化、并发、取消和安全改动扩大测试范围
- 全量与真实模型测试只用于高风险或发布批次
